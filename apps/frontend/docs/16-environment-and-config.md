# 16 — Environment & Config

A small footprint by design: one env var that matters, one Next config file,
one TypeScript config.

---

## Environment variables

Read at **build time** (anything in `process.env` is statically inlined for
client code; `NEXT_PUBLIC_*` is the safe prefix for browser-visible values).

| Variable                  | Default                                  | Used by                  | Notes                                 |
|---------------------------|------------------------------------------|--------------------------|---------------------------------------|
| `NEXT_PUBLIC_GENT_API_URL`| `https://gent-api.onrender.com/api`      | `src/lib/api-client.ts`  | Base URL for axios.                   |

That's it. Anything else you see in `process.env` came from outside this app
(Vercel, Render, etc.).

### `.env.example`

```bash
# Public API base URL. Leave empty to use the hosted instance.
NEXT_PUBLIC_GENT_API_URL=https://gent-api.onrender.com/api
```

Copy to `.env.local` and override locally:

```bash
cp .env.example .env.local
# point at a local Django backend
echo 'NEXT_PUBLIC_GENT_API_URL=http://localhost:8000/api' >> .env.local
```

### Adding a new env var

1. **Decide visibility.** Browser-visible? Prefix with `NEXT_PUBLIC_`.
   Server-only? Plain name. Never put secrets in `NEXT_PUBLIC_*` — they ship
   in the JS bundle.
2. **Add it to `.env.example`** with a default and a comment.
3. **Read it once** in a `src/lib/*.ts` module:

   ```ts
   export const NEW_THING = process.env.NEXT_PUBLIC_NEW_THING ?? "default";
   ```

4. **Document it** in the table above.

Do **not** read `process.env.X` directly from a page or component — it makes
unit testing and overrides painful.

## `next.config.ts`

```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  // intentionally minimal
};

export default nextConfig;
```

We keep it empty to avoid surprises. If you need something:

| Need                            | Add                                                   |
|---------------------------------|-------------------------------------------------------|
| Allow loading remote images     | `images.remotePatterns: [...]`                        |
| Rewrite/proxy a path            | `async rewrites() { return [...] }`                   |
| Custom headers (CSP, etc.)      | `async headers() { return [...] }`                    |
| Standalone build for Docker     | `output: "standalone"`                                |
| Increase the body parser limit  | per-route in the API route, not here                  |

Stay minimal. Every line added to this file is config we have to remember.

## `tsconfig.json`

Key settings:

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["dom", "dom.iterable", "esnext"],
    "module": "esnext",
    "moduleResolution": "bundler",
    "strict": true,
    "jsx": "preserve",
    "incremental": true,
    "plugins": [{ "name": "next" }],
    "paths": {
      "@/*": ["./src/*"]
    }
  },
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx", ".next/types/**/*.ts"],
  "exclude": ["node_modules"]
}
```

Things you should not change without a reason:

- `strict: true` — every project we ever loved had this on. Keep it.
- `paths: { "@/*": ["./src/*"] }` — the alias the whole codebase depends on.
- `target: ES2022` — Next 16 expects this.

    "@tailwindcss/postcss": {},

separately — Tailwind v4 includes its own prefixer.

## `eslint.config.mjs`

Uses the flat config format with `eslint-config-next`. Out of the box it
enforces:

- React + Next rules (no unused imports, missing keys, hooks rules).
- TypeScript rules from `@typescript-eslint`.

Run via:

```bash
npm run lint
```

If you need to disable a rule for a specific line, do it inline with a
comment and a brief reason — don't broaden the rule in the config.

## `components.json`

Generated by the shadcn CLI when primitives are scaffolded. Holds the
project's defaults for theme + paths. Not heavily used since most
primitives have been customised — but the file is kept so further scaffolds
land in the right place.

## What's intentionally **not** there

- **No `.env.production`** — production uses the same `NEXT_PUBLIC_GENT_API_URL`
  defaulted in code or injected by the host (Vercel/Render dashboard).
- **No `.babelrc`** — Next's SWC compiler is used; no Babel customisation.
- **No service worker / PWA config** — out of scope.
- **No analytics** — when we add one, prefix the env var with `NEXT_PUBLIC_`
  and add it to the table above.
